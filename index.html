시작하기 전에

먼저 3가지 용어부터 정리하기

  • 바이브 코딩 (Vibe Coding) — "느낌대로 짜고 AI가 알아서 해주길 바라는 것". AI에게 자연어로 부탁만 하면 코드가 나오는 시대의 코딩 방식
  • 에이전트 (Agent) — "스스로 생각하고 행동하는 AI 인턴". 질문에 답하는 챗봇과 달리, 목표를 주면 계획을 세우고 실제로 실행까지 함
  • 에이전틱 엔지니어링 (Agentic Engineering) — "AI 인턴이 잘 일하도록 환경과 규칙을 설계하는 기술". 바이브 코딩의 다음 단계

한 줄 요약 — 바이브 코딩은 "결과", 에이전트는 "주체", 에이전틱 엔지니어링은 "설계 기술"

02 / 15

개념 정리

AI 사용은 어떻게 진화해왔나

  • 1단계 — 프롬프트 엔지니어링 | ChatGPT/Gemini에게 "이렇게 답해줘" 하고 잘 묻는 기술. 대화 위주
  • 2단계 — 컨텍스트 엔지니어링 | AI에게 지침·데이터·도구를 함께 제공해서, 정해진 컨텍스트 안에서 최고 성능을 끌어내는 기술
  • 3단계 — 하네스 엔지니어링 | AI 코딩 에이전트가 오작동하지 않고 올바른 방향으로 일하도록, 에이전트를 둘러싼 외부 환경(컨텍스트 + 도구 + 검증 루프)을 설계하는 기술
"이제 우리는 프롬프트를 잘 쓰는 사람이 아니라, AI가 일할 환경을 설계하는 사람이 되어야 합니다."

모델 경제학

토큰과 모델 비용 이해하기

토큰(Token)이란? AI가 글을 읽을 때 쪼개는 최소 단위 — 한국어는 보통 1글자 ≈ 1~2토큰. "안녕하세요"가 약 5토큰. 모델은 이 토큰 수만큼 비용을 청구합니다.

모델입력 비용출력 비용주요 강점
Claude Opus 4.7$5.00/1M$25.00/1M자율성 극대화, 1M 컨텍스트
Gemini 3.1 Pro$2.00/1M$12.00/1M압도적 가성비, 구글 연동
GPT-5.5$2.50/1M$10.00/1M지시사항 지속성, 컴퓨터 유즈
DeepSeek V4-Pro$1.74/1M$3.48/1M오픈소스, 대량 작업 최적

💡 에이전트는 한 작업에 LLM을 수십~수백 번 호출합니다. 토큰 = 돈. 토큰을 아껴야 작업도 빨라지고 비용도 줄어요.

04 / 15

동작 원리

에이전트 = LLM + 도구 + 루프

🤖 일반 LLM (챗봇)

"질문 → 답변" 한 번으로 끝. 직접 파일을 만들거나 명령어를 실행하지 못함

⚙️ 에이전트

"목표 → 계획 → 행동 → 관찰 → 반복" 스스로 도구를 골라 쓰고, 결과를 보고 다음 행동을 결정

🛠️ 도구 (Tools)

파일 읽기/쓰기, 터미널 실행, 웹 검색 등 — AI가 손발처럼 쓰는 능력

🔁 루프 (Loop)

인간 개입 없이 "행동 → 관찰 → 수정"을 반복. 오류가 나도 스스로 고쳐가며 목표 달성

챗봇에게 "랜딩 페이지 만들어줘"는 코드만 보여주지만, 에이전트는 실제로 파일을 생성합니다.

05 / 15

핵심 도구

바이브 코딩을 잘하려면 이 5가지를 알아야 합니다

📂 컨텍스트 관리

AI가 한 번에 볼 수 있는 정보 영역(컨텍스트 윈도우)을 관리. 필요한 것만 보여주고 불필요한 건 비우기

🔌 MCP

Model Context Protocol. 외부 시스템(DB, Slack, Jira, 노션)과 AI를 연결하는 표준 콘센트

📚 Skill

반복되는 업무 절차(SOP)를 마크다운으로 패키징한 도구. "이런 요청이 오면 이렇게 처리해" 매뉴얼

👥 Sub-agent

큰 작업을 전문 에이전트들에게 나눠주기. 코드 리뷰어, 테스트 러너처럼 역할 분담

🛡️ 권한 설정 (Hook)

"어떤 행동은 허용, 어떤 행동은 금지"를 정하는 가드레일. 위험한 명령은 자동 차단

06 / 15

핵심 원리

컨텍스트 — AI의 단기 기억

🧠 컨텍스트란?

AI가 이번 대화에서 기억하는 모든 정보 (질문 + 답변 + 첨부 파일). 새 세션을 시작하면 모두 사라지는 단기 메모리

📏 모델마다 크기가 다르다

Claude Opus 4.7은 100만 토큰(책 ~10권), GPT-5.5는 30만, Gemini는 200만. 한 번에 담을 수 있는 양이 다름

⚠️ 많이 쌓이면 멍청해진다

컨텍스트가 가득 차면 AI가 핵심을 놓치고, 앞 내용을 잊거나 같은 실수를 반복함 ("Context Rot")

✂️ 그래서 관리가 필요

필요 없는 대화는 비우고, 핵심 정보만 남기기. 작업이 끝나면 새 세션으로 시작

💡 "AI가 갑자기 바보가 됐어"라고 느낄 때 → 보통 컨텍스트가 너무 쌓인 신호

하네스 기초

CLAUDE.md — 하네스 엔지니어링의 시작점

AI가 새 작업을 시작할 때마다 자동으로 가장 먼저 읽는 문서. "해야 할 것과 하지 말아야 할 것"을 정의하는 프로젝트 룰북입니다.

  • 🎯 하네스 엔지니어링의 기본 | AI 행동을 통제하는 가장 단순하고 강력한 도구
  • 🔁 매 세션마다 읽음 | 컨텍스트 시작에 항상 포함되므로 짧고 핵심적이게 작성해야 함
  • 📏 권장 분량 | 200줄 이내 (보통 60줄 내외). 길면 컨텍스트만 잡아먹고 효과 떨어짐
  • ✍️ 포함할 내용 | 코딩 컨벤션, 사용 금지 라이브러리, 빌드/테스트 명령어, "절대 ~하지 마라" 같은 규칙

📖 좋은 예시 → forrestchang/andrej-karpathy-skills (안드레아 카파시 본인의 CLAUDE.md 모음)

안전망

Git/GitHub — AI 시대의 안전망

🔄 변경 이력 추적

AI가 수정한 모든 내용을 기록. 언제든 되돌릴 수 있어 실수 걱정 없이 시도 가능

🤝 인간 검토 자리

AI는 세션 브랜치에서 작업, 인간은 PR(Pull Request)에서 최종 승인

☁️ GitHub Pages

코드를 올리기만 하면 자동으로 공개 웹사이트가 됨 (오늘 실습에서 사용)

  • commit = 변경 사항을 기록하는 "저장 지점"
  • push = 내 변경을 GitHub로 올리기
  • branch = 본 작업과 분리된 작업 공간
  • PR (Pull Request) = "내 변경 검토해주세요" 요청

일하는 방식

개발 워크플로우 — 4단계 사이클

에이전틱 엔지니어링도 결국 전통적인 개발 사이클을 따라갑니다. AI는 각 단계를 도와주는 동료입니다.

1 🗺️ 설계 (Planning) 무엇을 만들지 결정 — AI에게 요구사항 설명, 구조 합의 (/plan 모드 활용)
2 ⌨️ 구현 (Implementation) 실제 코드 작성 — AI가 코드 생성, 인간은 의도와 방향성 가이드
3 🐞 테스트 (Testing) 동작 검증 — AI가 테스트 작성·실행, 오류 발견 시 자동 수정
4 🚀 배포 (Deployment) 사용자에게 전달 — Git push → GitHub Pages, 이후 피드백 받아 다시 1번으로 반복

🔁 한 번에 완성되는 일은 없어요. 작은 사이클을 빠르게 여러 번 도는 게 핵심입니다.

실습

나만의 랜딩 페이지 만들고 공개하기

목표: AI에게 자연어로 부탁해서 랜딩 페이지를 만들고, 내 GitHub에 올려서 누구나 볼 수 있는 URL로 공개합니다.

1 프로젝트 폴더 작업할 빈 폴더를 만들고 IDE(Cursor/VSCode)로 엽니다
2 Gemini CLI 터미널에서 gemini를 실행해 에이전트를 켭니다
3 Skill 설치 디자인 가이드 스킬 가져오기 (awesome-design-md / ui-ux-pro-max-skill)
4 페이지 제작 "내 소개 랜딩 페이지를 만들어줘"처럼 자연어로 요청 → 에이전트가 HTML 생성
5 Git & GitHub git init 후 새 GitHub 저장소 만들기
6 커밋 & 푸시 코드를 GitHub에 업로드 (commit → push)
7 Pages 활성화 저장소 설정에서 Pages 켜면 → 🎉 공개 URL 완성!

핵심은 직접 코딩하지 않고, AI에게 "무엇을, 어디에, 어떻게"를 잘 설명하는 것입니다.
